# セキュリティ設計基準書

## 優先順位

| 優先度 | 項目 | 理由 |
|--------|------|------|
| 🔴 最重要 | DB側の RLS + REVOKE | データ漏洩の可能性につながるため |
| 🟠 重要 | Publishable Key をブラウザに露出しない | ※Publishable Key なので表に出しても問題はないが、見えないに越したことはない。DB防御の前段 |
| 🟠 重要 | AIエージェントAPIキー管理 | プロンプトインジェクションによる流出防止 |
| 🟡 必要 | 認証・トークン管理（Cookieベース） | なりすまし防止 |
| 🟢 補助 | CSRF対策・入力バリデーション | 多層防御 |

## 1. DB（Supabase）側のアクセス制御【最重要】

### 1-1. RLS（Row Level Security）の必須化

**必須設定：**

- 全テーブルで RLS を有効化（情報漏洩の可能性 → DBはユーザーアクセス以外は基本利用しない想定）
- anon ロールにはアクセスさせない
- authenticated ロールにも自分のデータのみ許可
- allowed_users テーブルに登録されたユーザーのみ操作可能

### 1-2. REVOKE（多層防御）をすること

- **GRANT** = 権限を与える
- **REVOKE** = 権限を剥奪する

**実施内容：**

```sql
REVOKE ALL ON TABLE allowed_users FROM anon;
REVOKE ALL ON TABLE allowed_users FROM authenticated;
```

**対象テーブル：**

| 種別 | アクセス方法 |
|------|-------------|
| allowed_users | Secret key を使った Admin クライアントのみ |
| 管理系テーブル | Secret key を使った Admin クライアントのみ |
| 課金関連テーブル | Secret key を使った Admin クライアントのみ |

### 1-3. テーブル設計原則

- 不要なテーブルを作らない
- テーブルが増えるほど RLS 設定漏れリスクが増える
- 複雑な権限周りの設定は避ける（例えばこのユーザーだけ特殊の扱いをする、非ログインユーザーの中でもこのドメインは〜など）

## 2. 環境変数・APIキー管理【重要】

### 2-1. NEXT_PUBLIC_ を利用しない

Next.js では以下の流れでフロントに露出する：

```
NEXT_PUBLIC_ 付き環境変数
  ↓
ビルド時にフロントJSへ埋め込み
  ↓
F12 > Sources で誰でも閲覧可能
```

※ `src/lib` 配下でも import された時点でブラウザに送られる。よくあるのが Supabase の公開キー。

**禁止事項：**

- フロントに API キーを置く
- `NEXT_PUBLIC_` に Supabase キーを入れる
- ブラウザに秘密情報を含める

**必須ルール：**

| 種類 | 保存場所 |
|------|----------|
| Publishable Key | サーバーサイドのみ |
| Secret Key | サーバーサイドのみ |

### 2-2. .env ルール

- `.env.local` は Git 管理しない
- `.gitignore` に含まれていることを確認
- Vercel デプロイ時は Vercel 環境変数設定から登録

**命名規則：**

| OK | NG |
|----|----|
| `SUPABASE_URL` | `NEXT_PUBLIC_SUPABASE_URL` |
| `SUPABASE_PUBLISHABLE_KEY` | `NEXT_PUBLIC_SUPABASE_ANON_KEY` |
| `SUPABASE_SECRET_KEY` | - |

## 3. 認証・トークン管理【必要】

### 3-1. 認証

- 認証なし API を作らない
- Supabase Auth（OAuth）を利用

### 3-2. トークン管理

| 方法 | 可否 |
|------|------|
| HttpOnly Cookie | ✅ 必須 |
| localStorage | ❌ 禁止 |

localStorage は以下の方法で取得可能なため、認証等で利用するトークン周りは必ず HttpOnly Cookie を採用すること。

```
F12 > Console
localStorage.getItem("token")
```

### 3-3. セッション管理

- `proxy.ts`（Next.js 16 の `middleware.ts` 後継）で全リクエスト前にトークンを自動リフレッシュ
- `@supabase/ssr` の `createServerClient` が Cookie 経由でトークンの読み書きを行う
- クライアント側にリフレッシュ処理は不要（proxy がサーバー側で透過的に処理）

## 4. フロントエンド設計ルール【必要】

**禁止：**

- Supabase クライアントをフロントで import
- ブラウザで直接 DB に接続

**正しい構成：**

```
Frontend ← Vercel でデプロイ
   ↓
fetch("/api/...")
   ↓
Server  ← Vercel でデプロイ
   ↓
Supabase
```

## 5. API Route セキュリティ【補助】

### 5-1. CSRF対策

- 必須（POST / PUT / DELETE）
- CSRF トークン導入
- `X-Requested-With: XMLHttpRequest` 検証
- ログアウト API にも適用

### 5-2. 認証チェック

- 全 API の冒頭で認証確認
- 共通関数 `authenticate()` を使用
- 各ルートに個別実装しない

### 5-3. 入力バリデーション（推奨）

- ホワイトリスト方式

## 7. AI開発環境・エージェント機能【重要】

### 7-1. 開発者ツール（Claude Code）【導入方針・検討中】

- **方針**: 各メンバーに Claude Code を導入する方向で検討中
- 各メンバーは Claude Max プラン（定額）を個人ログインで利用予定
- APIキー不要、ローカルにシークレットなし
- ※ プラン・運用ルールは確定次第更新する

### 7-2. アプリ組み込みAIエージェント（Gemini）

- SDK: `@google/genai`（環境変数で自動切り替え、コードに分岐なし）
- 開発中: Vertex AI（gcloud IAM認証、APIキー不要）
- 本番: Gemini APIキー（Vercel環境変数にのみ保存）

**環境変数のみで切り替え：**

```bash
# ローカル .env.local
GOOGLE_GENAI_USE_VERTEXAI=true
GOOGLE_CLOUD_PROJECT=your-project-id

# 本番 Vercel環境変数
GOOGLE_GENAI_USE_VERTEXAI=false
GEMINI_API_KEY=xxx
```

**コードは環境を意識しない：**

```typescript
// lib/ai.ts — これだけ
import { GoogleGenAI } from '@google/genai'
export const ai = new GoogleGenAI()
```

SDKが `GOOGLE_GENAI_USE_VERTEXAI` を自動で確認し、`true` の場合はgcloud IAM認証でVertex AIに、`false` の場合は `GEMINI_API_KEY` でGemini APIに接続する。

### 7-3. BigQuery認証

- 開発中: gcloud IAM認証（`gcloud auth application-default login`）
- 本番: サービスアカウントJSON（Vercel環境変数）
- 顧客ごとにサービスアカウントを分離、対象データセットのみ権限付与

**ローカル開発は1コマンドのみ：**

```bash
gcloud auth application-default login
# → Vertex AI も BigQuery も同じ認証で動く
```

### 7-4. ゼロトラスト構成まとめ

「何も信頼しない、常に検証する」原則に基づく多層防御。

| レイヤー | 対策 | 考え方 |
|----------|------|--------|
| 開発環境 | gcloud IAM認証 | APIキー自体を排除（流出するものがない） |
| 本番APIキー | Vercel環境変数のみ | 開発者が直接触れない |
| DB | RLS + REVOKE | 認証済みでも最小権限 |
| BigQuery | サービスアカウント + データセット分離 | 顧客ごとに権限を限定 |
| アプリ | Supabase Auth（HttpOnly Cookie） | トークンをブラウザに渡さない |
| API Route | 全冒頭で認証チェック | 内部でも信頼しない |

## 8. コスト管理・監視

### 8-1. 一括請求構成

| サービス | 仕組み |
|----------|--------|
| GCP（Vertex AI / BigQuery / Gemini API） | Billing Account → 複数プロジェクトを紐付け |
| Vercel | Team → 複数プロジェクト |
| Supabase | Organization → 複数プロジェクト |

### 8-2. 顧客ごとのコスト把握（GCP内）

GCPリソースにラベルを付与することで、Billing Dashboardで顧客ごとに集計できる。

```typescript
// BigQueryクエリ実行時
const [rows] = await bigquery.query({
  query: `SELECT * FROM ...`,
  labels: { client: 'client_a' }
})
```

- GCP Billing Dashboardで `client` ラベルごとに集計可能
- GCP Budgetアラートで顧客ごとに上限設定可能

### 8-3. 異常検知・監視方針

- Cloud Audit Logsで誰がいつ何を叩いたかを記録
- 急激なAPI呼び出し増加 → 漏洩・不正利用の検知
- キーローテーションより「漏れない構造」を優先
- 異常検知 + 即時対応フロー（キーの即時無効化）を整備しておく
